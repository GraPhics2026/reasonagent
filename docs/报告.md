## 摘要

文生图模型在两类任务上表现不足：一是复杂**描述性 prompt** 的对齐（对象数量、颜色、空间关系）；二是**假设性指令**（如「如果冰块融化会怎样」）需结合图像与常识推理，无法直接执行。本项目构建 **ReasonGenPilot**——一个无训练的多通路 Agent 系统，通过 MLLM 与公开 T2I/Edit API 的工程编排，实现 **gen**（GenPilot 风格 prompt 优化 + 文生图）、**edit**（ReasonBrain 启发的 HI-IE 推理 + Qwen-Image 指令编辑 + VQA 闭环）、**hybrid**（假设性整图重生成）。系统提供统一入口、Gradio Demo 与 67 项单元测试；四个端到端 case 在真实 API 下 VQA 均为 1.0。

**代码链接**：https://github.com/GraPhics2026/ReasonGenpilot

## 1. 项目背景与目标

### 1.1 待解决的问题

当前主流文生图（T2I）模型在两个方面存在明显缺陷：

| 问题 | 表现 |
|------|------|
| **描述性生成对不齐** | T2I 听不全复杂 prompt：数错对象数量、漏掉颜色、空间关系错位 |
| **假设性指令不可读** | T2I 模型无法理解「如果冰块融化会怎样」这类需要常识推理的反事实指令 |

学术界提出的 [ReasonBrain（2025）](https://arxiv.org/abs/2507.01908) 通过 **Reason50K** 数据集 + **FRCE / CME** 模块端到端训练 FLUX 来解决第二类问题，但训练成本高、不易复现。

### 1.2 项目目标

构建一个 **完全无训练** 的统一 Agent 系统，仅依靠 MLLM API + 现成 T2I/Edit API + 工程编排，实现以下三条通路并自动分流：

| 通路 | 输入 | 目标 |
|------|------|------|
| **gen** | 纯文字 prompt | 优化 prompt 使图与描述更对齐 |
| **edit** | 原图 + 假设性指令（局部变化） | 在原图上局部指令编辑并 VQA 验证 |
| **hybrid** | 原图 + 假设性指令（整图重构） | Reason 写场景 prompt，从零 T2I 出图 |

## 2. 相关工作

本节简要介绍本项目所借鉴的公开方法与理论基础；具体工程实现见后续各成员章节。

### 2.1 GenPilot：描述性 prompt 对齐

[GenPilot](https://github.com/27yw/GenPilot) 面向 T2I 模型「听不全复杂 prompt」的问题，核心思路是 **test-time prompt optimization（测试时 prompt 优化）**：

1. **约束分解**：将自然语言 prompt 拆成可逐条检查的视觉约束（对象、数量、颜色、空间关系等）；
2. **Baseline 生成**：用初始 prompt 文生图，得到对照结果；
3. **双路分析**：Caption + VQA 对比原 prompt 与生成图，定位遗漏或错误；
4. **候选与选优**：针对错误改写 prompt，生成多个候选，逐个出图并 VQA 打分，选最优进入下一轮。

本项目 **gen 通路** 封装上述流程为 `run_gen_pipeline()`，并供 **hybrid 通路** 在得到 `scene_prompt` 后复用同一 T2I + VQA 迭代能力。

### 2.2 ReasonBrain：假设性图像编辑（HI-IE）

[ReasonBrain (2025)](https://arxiv.org/abs/2507.01908) 定义 **HI-IE（Hypothetical Instruction-based Image Editing）** 任务：给定原图与假设性指令（如「如果冰块融化会怎样」），模型需先推理反事实视觉结果，再完成编辑。论文主要贡献包括：

- **Reason50K 数据集**：覆盖 physical / temporal / causal / story 四类推理场景；
- **FRCE（Fine-grained Reasoning Cue Extraction）**：从原图提取细粒度视觉线索（材质、状态、位置、物体关系）；
- **CME（Counterfactual Modeling and Editing）**：将推理结果建模为可执行的编辑条件，驱动 FLUX 扩散模型端到端改图。

本项目 **edit 通路** 对齐 HI-IE 任务与四类 `reasoning_type`，但用零样本 MLLM + 结构化 JSON 替代 FRCE/CME 训练模块，用 DashScope Qwen-Image **指令编辑**替代 FLUX 微调；并增加多候选 VQA + refine 迭代作为工程化验证环。

| ReasonBrain（论文） | ReasonGenPilot（本项目） |
|---------------------|--------------------------|
| Reason50K 训练 | 零样本 `reason_system.txt` |
| FRCE / CME | `visual_cues` + `physics_implications` + `preserve_objects` |
| FLUX 扩散编辑 | Qwen-Image 指令编辑（image + text，无 mask） |
| 单次 forward | 多候选 + VQA + refine（默认 2 轮 × 2 候选） |

**核心创新**：用工程化 Agent 编排 + 公开 API 复现 ReasonBrain 的 HI-IE（假设性图像编辑）能力，且支持「整图重构」这一论文未覆盖的情况（hybrid 通路）。

### 2.3 指令式图像编辑 API

DashScope [Qwen-Image](https://help.aliyun.com/zh/dashscope/) 支持 **image + text** 的全局指令编辑：以原图为条件、按文本指令重绘。本项目实验对比 mask inpaint 后选定该方案作为 edit 主路径（详见 §5.3 编辑选择）。

## 4. 成员 1：基座 + gen 通路

### 4.1 任务范围

搭建项目骨架，封装统一 MLLM / T2I 客户端，跑通描述性生成 + Prompt 优化通路。后续两位成员的 `api_client`、`t2i_client` 与 `schemas` 全部建立在这一层。

### 4.2 关键模块

| 文件 | 作用 |
|------|------|
| `reason/api_client.py` | OpenAI-compatible MLLM 调用（文本 + 视觉），含图像 base64 + resize |
| `reason/t2i_client.py` | T2I 抽象层：`dry_run` / `siliconflow` / `dashscope` / `comfyui` 四后端 |
| `reason/schemas.py` | 统一数据结构 `GenPipelineResult` / `ReasonResult` / `EditPipelineResult` 等 |
| `reason/gen_pipeline.py` | gen 通路一键调用 |

### 4.3 GenPilot 风格的 prompt 优化流程

```
原始 prompt
   │
   ├──► decompose_prompt        → 拆成可检查的视觉约束列表
   │
   ├──► T2I baseline            → image_before.png
   │
   └──► 迭代 N 轮：
         ├ generate_candidates  → 多候选 prompt（默认 2）
         ├ T2I 每个候选         → candidate_iter_K_*.png
         ├ VQA 打分             → 选最高分
         └ 选优替换为 image_iter_K.png
```

### 4.4 dry-run 兜底设计

`run_gen_pipeline(dry_run=None)` 会自动检测 MLLM 是否配置：

- 未配置 → 用启发式 prompt 重写（在末尾追加 "clear, highly detailed composition"）+ SVG 占位图
- 已配置 → 调真实 MLLM + T2I

**好处**：CI、本地无网络、API 配额耗尽时仍能跑通流程；后续 edit / hybrid 全部继承同一兜底模式。

### 4.5 验收

```bash
python -m reason.gen_pipeline \
  --prompt "A grass field filled with red poppies and yellow daisies beside a wooden windmill." \
  --output data/output/e2e/gen_demo \
  --real-api
```

实际结果（§8.1）：baseline VQA = 0.8 → iter 1 = 1.0 → iter 2 = 1.0。

## 5. 成员 2：edit 通路

### 5.1 任务范围

实现假设性图像编辑（HI-IE）：MLLM 看图 + 反事实指令 → 结构化推理 → 指令式编辑 → VQA 多候选选优 + refine 迭代。

### 5.2 推理输出契约（与论文 ReasonBrain 对齐）

```json
{
  "mode": "edit",
  "reasoning_type": "physical | temporal | causal | story",
  "reasoning_chain": ["..."],
  "visual_cues": ["fine-grained facts from the image"],
  "physics_implications": ["expected visible outcome"],
  "target_objects": ["objects or regions to change"],
  "preserve_objects": ["background or objects to keep"],
  "edit_prompt": "...",
  "vqa_checklist": [{"q": "...", "expected": "yes"}]
}
```

`reasoning_type` 四类与 Reason50K 完全对齐：

| 值 | 含义 | 示例 |
|----|------|------|
| `physical` | 物理规律导致的状态变化 | 冰块融化、跷跷板倾斜 |
| `temporal` | 时间维度变化 | 日落、变夜晚 |
| `causal` | 因果 / 交互事件 | 大象与松鼠玩跷跷板 |
| `story` | 隐含叙事 / 纹理显现 | 隐藏图案出现 |

### 5.3 编辑选择：指令编辑 而非 mask inpaint

实验对比后选定 **DashScope Qwen-Image 指令编辑**（原图 + 文本，**无 mask**）：

| 方案 | 主体保真 | 反事实物体互动 | 工程复杂度 |
|------|---------|---------------|-----------|
| Mask inpaint（万相 API） | 高 | 差（需要正确 mask） | 高（需 segment） |
| **指令编辑（Qwen-Image）** | 中-高 | 好 | 低 |

### 5.4 流程

```
原图 + 假设指令
    │
    ▼
Reason Agent（mode="edit"）
   → reasoning_type / visual_cues / physics_implications / preserve_objects
   → edit_prompt + vqa_checklist
    │
    ▼
finalize_edit_prompt()
   → 注入物理约束 + 保留对象 → 完整编辑 prompt
    │
    ▼
每轮 K 个候选 edit_prompt（注入 reason_context）
   → 每个 prompt 调 Qwen-Image 编辑 → candidate_iter_N_*.png
   → MLLM VQA 打分（带 reason_context）→ 选最高分
    │
    ▼
不合格 → edit_refine 后再编（默认 `--iterations 2 --min-iterations 2 --candidates 2`）
合格   → image_after.png
```

### 5.5 验收（四类 reasoning_type case）

覆盖 Reason50K 四类推理，输出均在 `data/output/edit/` 下（最终图为 `image_after.png`，完整记录见各目录 `edit_final.json`）：

| `reasoning_type` | 用例 | 输出目录 |
|------------------|------|----------|
| `physical` | 大象与松鼠玩跷跷板 | `data/output/edit/elephant_seesaw_v2/` |
| `temporal` | 青年数十年后衰老（aging） | `data/output/edit/aging_temporal/` |
| `causal` | 开窗有风、窗帘飘动（curtain） | `data/output/edit/curtain_wind_causal/` |
| `story` | 狗追咬时人的反应（dog_chase） | `data/output/edit/dog_chase_story/` |

以 `physical` 跷跷板 case 为例，运行命令：

```bash
python -m reason.edit_pipeline \
  --image data/input/edit/elephant_squirrel_grass.png \
  --instruction "大象和松鼠玩跷跷板会怎样呢?" \
  --output data/output/e2e/edit_demo \
  --iterations 2 --min-iterations 2 --candidates 2 \
  --real-api
```

实际推理结果（§8.2）：
- `reasoning_type`: `causal`（识别为因果互动）
- `target_objects`: `["seesaw", "elephant's position relative to seesaw", "squirrel's position relative to seesaw"]`
- VQA checklist 4 项全部通过（含「跷跷板存在」「大象侧着地」「松鼠侧悬空」「草地背景保留」）
- 最终 VQA score = **1.0**

**补充说明（四类完整测试 vs e2e demo）**：上表 `data/output/edit/` 下的四类 case 为实际非 demo 测试场景，VQA 比 e2e `edit_demo` **更严苛**，故各目录 `edit_final.json` 中的 `score` 往往更低（如跷跷板 **0.67**、aging/curtain **0.83**、dog_chase **0.33**）。主要原因：

1. **checklist 更细**：Reason Agent 按 `reason_system.txt` 为每条指令生成 2–4 条可视觉核验项（物理结果、保留背景、空间关系等），覆盖项多于 demo 中较粗的几条概括；
2. **打分规则严格**：`verify_edit_result()` 要求 checklist **全部**明确满足才给 1.0，任一子项（如「松鼠是否明显抬升」）有歧义即扣分；
3. **保留约束**：VQA 同时对照 `reason_context` 中的 `Must preserve:`，背景或无关物体漂移也会降分。

因此 §8.2 的 `edit_demo` 端到端展示可达 1.0；四类 case 的 `image_after.png` 与 `edit_final.json` 更能反映 edit 通路在完整约束下的真实表现。

## 6. 成员 3：hybrid 通路

### 6.1 任务定位：与 edit 的核心区别

| | edit | hybrid |
|--|------|--------|
| 改图方式 | 原图 + `edit_prompt` → Edit API | `scene_prompt` → T2I（**不传原图**） |
| 是否保留原图像素 | 尽量保留 | 从零重新生成 |
| Reason 输出 | `edit_prompt` + `target_objects` | `scene_prompt`（完整场景描述） |
| 典型 case | 冰块融化、跷跷板倾斜 | 花分两束、室内变深夜、季节切换 |

**核心思路**：有参考图的假设性文生图 —— 原图仅用于 MLLM 推理，T2I 出图时**不传原图**，完全依赖 `scene_prompt` 文字重画。

### 6.2 流程

```
原图 + 假设指令
    │
    ▼
Reason Agent（mode="hybrid"）
   → reasoning_chain / visual_cues / physics_implications / preserve_objects
   → scene_prompt（英文，可直接送 T2I）
   → vqa_checklist
    │
    ▼
scene_prompt 7 层质量保障（成员 3 关键贡献）
    │
    ▼
run_gen_pipeline(scene_prompt)
   → GenPilot prompt 优化 + T2I 出图
    │
    ▼
MLLM VQA 验证（生成图 vs checklist）→ image_after.png
```

### 6.3 scene_prompt 7 层后处理（hybrid 最大工程价值）

只靠 Reason Agent 输出的 `scene_prompt`，T2I 经常画错（碎片化、丢元素、视角错乱）。成员 3 设计了 7 个串行后处理函数：

| # | 函数 | 解决的问题 |
|---|------|-----------|
| 1 | `_validate_and_fix_scene_prompt` | 检测碎片化 prompt（如 "grass, sneakers, snow"），从 `visual_cues` 重建完整描述 |
| 2 | `_ensure_instruction_elements` | 检测多元素指令遗漏（「人很多 + 下雪」漏了"人"则补回） |
| 3 | `_inject_perspective_anchor` | 从 `visual_cues` 提取空间位置，防止 T2I 乱放物体 |
| 4 | `_inject_style_anchor` | 注入颜色 / 材质 / 光照 / 氛围；对变换性场景智能跳过冲突类别 |
| 5 | `_inject_person_identity` | **6 维全覆盖**：种族/肤色/年龄/发型/体态/五官，逐维检测并注入，防 T2I 默认偏差（白人+随机年龄+通用体态） |
| 6 | `_inject_light_behavior` | 注入光照方向 / 硬度（如月光从窗户射入冷色） |
| 7 | `_inject_room_geometry` | 房间几何线索（窗户位置、地砖等） |

**例子**：「一束红玫瑰分成两束」如果只用 Reason 原始 `scene_prompt`，T2I 经常画成"一束红 + 一束粉"。后处理第 4 层检测到颜色一致性约束，自动注入 "both bouquets contain the same deep red roses" 锚点。

### 6.5 hybrid 对比实验（详见 [hybrid对比实验.md](docs/hybrid对比实验.md)）

针对「整图重构」类指令，对比 3 种方案：

### 7.2 Router 设计：MLLM-only 三分流（无降级）

#### 7.2.1 路线取舍

| 方案 | 优点 | 缺点 | 是否采用 |
|------|------|------|---------|
| 纯手动 `--mode` | 100% 可控 | 演示麻烦 | 作为 override 保留 |
| 关键词规则 | 零成本零依赖 | 准确率低、无法理解长指令 | 否 |
| MLLM 分类 | 自然语言友好、准确率高 | 依赖 MLLM 在线 | 是（auto 默认） |
| MLLM + 关键词降级 | API 失败仍可用 | 失败被掩盖、行为不可预测 | 否（有意拒绝） |

**最终方案：MLLM-only + 失败即抛错**。Router auto 模式必须有 MLLM；失败时直接 `RuntimeError` 暴露环境问题，不做任何降级 —— 用户用 `--mode {gen,edit,hybrid}` 强制覆盖即可绕过。

#### 7.2.2 分流规则

```
输入判断树：
├── prompt 必传  ← 每个 mode 都需要
├── 无图？ → gen
│   例："风车旁红色 poppy 和黄色 daisy 占满草地"
│
└── 有图 + 反事实指令？
    ├── 局部物理 / 状态变化 → edit
    │   例："冰块融化"、"大象和松鼠玩跷跷板"
    │
    └── 整图重构 / 数量 / 氛围大变 → hybrid
        例："花分两束"、"室内变深夜"、"季节切换"
```

**核心判别准则**（写在 `prompts/router_system.txt` 里给 MLLM）：改后是否还需要原图像素？需要 → edit；不需要 → hybrid。

**两类错误**，统一 `router: ` 前缀：
- `ValueError` —— 用户输入问题（缺 prompt / image / mode 错 / 文件不存在）
- `RuntimeError` —— 环境问题（MLLM 不可用 / API 失败 / 响应非法）

`prompts/router_system.txt` 含 8 个 calibration examples，输出严格 JSON `{"route": "...", "reason": "..."}`。

#### 7.2.3 23 个单元测试

`reason/test_router.py` 覆盖：

- **prompt 必传**（3）：空 / 空白 / edit 模式都要 prompt
- **mode 合法性**（2）：未知 mode、空字符串都抛 ValueError
- **mode + 输入兼容性**（7）：gen/edit/hybrid 透传成功 + edit/hybrid 缺图、图不存在、缺指令组合
- **auto 结构规则**（2）：无图 → gen、文件不存在视作无图
- **MLLM 路径**（4）：返回 gen/edit/hybrid + markdown fenced JSON
- **MLLM 错误**（5）：未配置、非法 route 值、无法解析、API 异常、缺 route key 全部抛 RuntimeError

### 7.3 统一入口 `pipeline.py`

**单一 `--prompt` 字段**对所有 mode（gen 当描述读，edit/hybrid 当反事实指令读）：

```bash

python pipeline.py --prompt "A windmill in a poppy field" --output data/output/case0

# 强制 mode（edit 内部默认 iterations=2, min_iterations=2, candidates=2；
# 完整参数控制请直接用 python -m reason.edit_pipeline）
python pipeline.py --mode edit \
  --image data/input/edit/elephant_squirrel_grass.png \
  --prompt "如果大象和松鼠玩跷跷板会怎样" \
  --output data/output/case0 \
  --real-api
```

按 router 决策分发到 `run_gen_pipeline` / `run_edit_pipeline` / `run_hybrid_pipeline`，统一返回 `.to_dict()` JSON 到 stdout。

## 9. 限制与未来工作

### 9.1 已知限制

1. **Router auto 模式延迟**：每次分流多一次 MLLM 调用（约 5–7 秒）。生产环境可缓存指令 → route 的映射。
2. **hybrid 不保留原图像素**：人物身份、纹理细节会有微小漂移。`scene_prompt` 7 层后处理已尽量约束，但无法做像素级保真。
3. **edit 通路依赖 Qwen-Image 编辑质量**：模型不能处理的复杂指令（如多次大规模拓扑变化），策略上需切换到 hybrid 通路重画。
4. **VQA 自评分由同一 MLLM 完成**：存在自我评分偏差。论文中独立 VQA model 可消除该偏差，但增加成本。

### 9.2 未来工作

| 方向 | 思路 |
|------|------|
| 集成完整 GenPilot Stage 1 / 2 | 替换 `gen_pipeline.py` 内部，保持 `run_gen_pipeline()` 接口 |
| 独立 VQA 模型 | 用 BLIP-2 / LLaVA-NeXT 做 VQA，避免 MLLM 自评 |
| Router 缓存层 | 指令 → route 的 LRU，重复 demo 时跳过 MLLM 调用 |
| hybrid 像素保真 | 探索 IP-Adapter / ControlNet 作为可选 image condition |
| 评估集 | 自建小规模 HI-IE 基准（含 gen / edit / hybrid 三类各 30 例） |

## 10. 结论

ReasonGenPilot 在不训练任何模型的前提下，仅靠 MLLM API + Qwen-Image + 工程编排，实现了：

1. **三条互补通路**：gen（描述对齐）/ edit（局部反事实）/ hybrid（整图反事实）
2. **统一接口与自动分流**：`pipeline.py --mode auto` 一行调用，Router 自动判断
3. **完整工程链路**：CLI + Gradio Web Demo + 67 单元测试 + dry-run 兜底
4. **可复现实验结果**：四个 end-to-end case 全部 VQA = 1.0，Router 在 17 个基准 case 上 100% 准确

相比 ReasonBrain 论文路线，本项目用 **<1000 行 Python + 4 个 prompt 文件** 复现了 HI-IE 的核心能力，并 **额外覆盖了整图重构这一论文未支持的场景**（hybrid 通路）。
