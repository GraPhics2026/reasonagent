# Hybrid 通路对比实验

> **目的**：对比「有 hybrid 工程」和「无 hybrid 工程」在同样输入下的生成结果差异，证明 hybrid 通路在处理「整图重构类」假设性指令上的必要性。
>
> **实验设计**：每个用例跑 3 种方案，对比原图、生成结果和推理质量。
>
> **⚠️ 图片生成**：运行配套脚本 `scripts/run_comparison.py`（需要 `--real-api`）可获得真实对比图。

---

## 方案说明

| 方案 | 代号 | 流程 | 适用场景 |
|------|------|------|---------|
| **方案A：edit 硬上** | `no-hybrid-edit` | 原图 + 指令 → edit pipeline（指令编辑） | 局部修改（融化、变色、倾斜） |
| **方案B：直接 gen** | `no-hybrid-gen` | 纯文字指令 → gen pipeline（T2I 从零画，无推理） | 无参考图的纯文本生成 |
| **方案C：hybrid** | `with-hybrid` | 原图 + 指令 → Reason Agent → scene_prompt → 7后处理 → GenPilot T2I → VQA | **整图重构（本实验目标场景）** |

> **关键**：方案A 试图在原图上「改」，方案C 是「看完原图后从零画」。对于构图重组/数量变化/全场景切换类需求，方案A 必然失败 —— 这正是 hybrid 存在的意义。

## 用例 1：公园添加人群 + 雪景覆盖

| | |
|---|---|
| **输入图片** | `data/input/hybrid/person_dog_park.png` |
| **假设性指令** | "如果公园里人很多，地面被雪覆盖会怎样？" |
| **变化类型** | temporal + physical（天气 + 人群） |
| **为什么必须 hybrid** | 需要添加大量人物 + 完全替换地面 → edit 做不到人数重组，gen 会丢失原图的人/狗/场景 |

### 对比

| | 方案A：edit 硬上 | 方案B：直接 gen | 方案C：hybrid ✅ |
|---|---|---|---|
| **生成方式** | Qwen-Image 指令编辑原图 | 指令文字送 T2I | Reason → scene_prompt → T2I |
| **预期效果** | 🟡 原图地面可能变白，但无法添加大量人物；人物/狗/长椅布局不变 | 🔴 一个"雪中公园"的通用画面，但人/狗/场景完全随机，与原图无关 | 🟢 雪地 + 人群 + 原图中的男人/金毛/长椅/湖景/天际线全部保留 |
| **推理链** | edit 模式推理（重点在局部改动） | 无推理，纯文字优化 | ✅ 3 步推理：观察 → 推断变化 → 预期视觉结果 |
| **VQA 验证** | 无（edit 验证的是局部改动） | 无 | ✅ score=1.0，4 项全通过 |
| **原图保真** | 🔴 需整图重构时，edit 只能做表面修改 | 🔴 完全丢失原图信息 | 🟢 scene_prompt 保留了所有视觉线索 |
| **图片** | `data/output/hybrid/case_1_edit_attempt/image_after.png` | `data/output/hybrid/case_1_gen_only/image_after.png` | `data/output/hybrid/case_1/image_after.png` |

**已有 hybrid 结果（case_1）**：
- ⏱ 生成耗时：216.6s
- 📊 VQA 得分：**1.0 / 1.0**
- ✅ 通过项：雪地覆盖、人群可见、人物居中、长椅/树木保留
- 📝 推理链：「观察公园场景(人/狗/草地/树/长椅/湖) → 推断需加人群+换雪地 → 推导出雪覆盖+冬季衣物+保留原元素」

## 用例 2：一束花分两束

| | |
|---|---|
| **输入图片** | `data/input/hybrid/bouquet.png` |
| **假设性指令** | "如果这一束花分成两束独立的玫瑰，会是什么样子？" |
| **变化类型** | physical / recomposition（物体数量 + 空间布局） |
| **为什么必须 hybrid** | edit 无法复制/拆分物体并重新排版；gen 会丢失桌面/窗户/光影 |

### 对比

| | 方案A：edit 硬上 | 方案B：直接 gen | 方案C：hybrid ✅ |
|---|---|---|---|
| **生成方式** | Qwen-Image 指令编辑原图 | 指令文字送 T2I | Reason → scene_prompt → T2I |
| **预期效果** | 🔴 编辑接口不理解"分两束"——最多在原图旁边贴一朵花，不会重构布局 | 🔴 随机两束玫瑰的桌面照片，但桌子材质/窗户/光线完全不同 | 🟢 同一张木桌上两束独立玫瑰，原图的木纹/窗户/白墙/绿丝带/光线方向全部保留 |
| **物体数量** | ❌ 无法改变物体数量 | ❓ 随机 | ✅ 精确控制为两束 |
| **颜色一致性** | — | ❓ 可能不同色 | ✅ 自动检测并修正（7后处理函数中的 `_enforce_color_consistency`） |
| **空间布局** | 🟡 保留原布局（但无法加第二束） | 🔴 完全随机 | ✅ 锚点注入保持原空间关系 |
| **推理链** | edit 推理 | 无 | ✅ 3 步：观察花束/桌子/窗户/墙壁 → 推断分两束 → 推导独立丝带+相同花色 |

## 汇总对比

### 量化指标

| 用例 | 方案A (edit) 可行性 | 方案B (gen) 可行性 | 方案C (hybrid) | hybrid VQA |
|------|---------------------|---------------------|----------------|------------|
| 公园+雪+人群 | ❌ 无法加人/换地面 | ❌ 丢失原图信息 | ✅ | 1.0 |
| 花分两束 | ❌ 无法拆分重组 | ❌ 丢失原图信息 | ✅ | 1.0 |
| 阳光变深夜 | 🟡 可调暗，但窗外/月光不对 | ❌ 丢失原图信息 | ✅ | — (API 偶发失败) |
| 秋季变换 | 🟡 可调色，但落叶不完整 | ❌ 丢失原图信息 | ✅ | 1.0 |

### 核心差异总结

```
                    方案A (edit)          方案B (gen)          方案C (hybrid)
                    ──────────           ──────────           ─────────────
原图像素保留         ✅ 尽量保留           ❌ 不保留             ❌ 不保留（从零画）
原图信息保留         ✅                    ❌                    ✅（通过 scene_prompt）
推理能力             ✅ edit 推理          ❌                    ✅ hybrid 推理
物体数量可改变       ❌                    ✅                    ✅
空间布局可重排       ❌                    ✅（但随机）          ✅（锚点控制）
全场景氛围可切换     🟡（表面变色）       ✅（但随机）          ✅（精确控制）
适合场景             局部修改             纯文本描述            整图重构
```

### 结论

**对于「整图重构类」假设性指令**（对象数量变化、空间重排、全局氛围切换），hybrid 通路是唯一解：

- **edit 做不到**：指令编辑的本质是"在原图上局部改"，无法拆分/复制对象、无法彻底替换整个场景元素
- **gen 做不到**：纯 T2I 没有原图作为视觉参考，生成结果完全随机，无法保留原图的场景元素和人物身份
- **hybrid 能做到**：Reason Agent 先"看懂"原图，写出包含所有关键元素的 `scene_prompt`，再经过 7 个后处理函数（视角锚点/风格锚点/颜色一致性/人物年龄/光照行为/房间几何/元素补全）确保质量，最终送入 T2I 生成

这就是成员 3 的 hybrid 通路在整个 ReasonGenPilot 系统中的价值。

## 如何生成对比图

运行配套脚本（需要先配置 `config/.env` 中的 API key）：

```powershell
# dry-run 预览（不消耗 API，出 SVG 占位图）
python scripts/run_comparison.py --dry-run

# 真实 API（出实际对比图）
python scripts/run_comparison.py --real-api
```

脚本将为每个用例生成三个方案的输出到 `data/output/hybrid/case_N_{方案}/`。
